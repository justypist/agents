## Context

当前聊天页面使用 `@ai-sdk/react` 的 `useChat` 直接向 `/api/[agentId]/[sessionId]` 发起流式请求。服务端在 `streamChatSessionTurn` 中将 `request.signal` 传给 agent，并只在流结束的 `onFinish` 中保存完整消息。因此浏览器关闭、刷新或网络断开会取消请求，导致 agent 执行中断，数据库也可能只保留旧消息。

本变更需要把“接受用户消息”和“执行 agent 回复”拆开：请求一旦被服务端接受，就先持久化用户消息和后台执行状态；agent 回复由服务端后台任务继续完成，前端只负责观察状态和展示已有结果。

## Goals / Non-Goals

**Goals:**
- 用户消息被接受后，即使前端关闭或断开，agent 回复仍能继续执行并最终写入会话。
- 会话页面重新打开后能从数据库恢复消息、运行中状态、完成结果或失败信息。
- 同一条已接受消息不得因重试、刷新或重连被重复执行。
- 后台执行失败时保留失败状态，并向用户展示清晰错误。
- 尽量复用现有 agent 解析、skill 调用、消息校验和标题生成逻辑。

**Non-Goals:**
- 不实现跨多台服务实例的分布式队列调度，除非现有部署环境已经提供可用机制。
- 不改变 agent 的回复内容策略、skill 调用语义或附件上传流程。
- 不要求前端关闭期间实时推送；重新打开后可通过查询或轮询恢复状态。

## Decisions

1. 使用持久化执行状态作为恢复来源，而不是依赖浏览器流连接。

   - 在数据库中记录当前会话 turn 的状态，例如 `idle`、`running`、`completed`、`failed`，并记录正在处理的 user message id、错误信息和更新时间。
   - 接受新消息时先把用户消息写入会话，再将状态置为 `running`，之后启动后台执行。
   - 选择此方案是因为数据库已经是会话消息来源，重新打开页面天然可以读取状态；相比只在内存中继续 Promise，更能覆盖刷新和短暂断线。

2. 后台执行不得绑定 `request.signal`。

   - 当前 `streamChatSessionTurn` 将请求 abort 信号传给 agent，这是断开即中止的直接原因。
   - 新后台执行服务应独立于 HTTP 请求生命周期运行，并在完成或失败时更新数据库。
   - 如果运行环境无法保证普通异步任务在请求返回后继续执行，应优先接入项目已有或部署平台支持的后台任务机制，而不是假设 Node 进程内任务一定可靠。

3. 前端发送接口改为“提交 turn”，前端恢复接口读取会话快照。

   - 提交接口返回已接受状态和当前会话快照，不再要求保持流连接直到 agent 完成。
   - 聊天页在状态为 `running` 时展示“正在后台回复”，并通过轮询或页面刷新读取最新会话。
   - 这会牺牲断开后的实时 token 流，但换取关闭页面后的完成可靠性；页面打开期间仍可后续再补充实时体验。

4. 使用幂等键避免重复执行。

   - 以最后一条用户消息 id 或显式 client request id 作为当前 turn 的幂等标识。
   - 当同一会话已有相同 user message id 的 `running` 或 `completed` 记录时，重复提交只返回现有状态，不再次调用 agent。
   - 当另一个 turn 正在运行时，新提交应被拒绝或排队；本变更优先采用拒绝并提示等待，以减少并发合并消息的复杂度。

5. 保存失败状态并保留用户消息。

   - agent 执行异常时将状态置为 `failed`，记录错误摘要，不删除已提交的用户消息。
   - 用户重新打开页面后可以看到失败提示，并可选择重新发送或后续新增重试能力。

## Risks / Trade-offs

- [后台任务可靠性不足] → 如果部署平台会在响应结束后冻结运行时，需要使用平台后台任务、队列或持久工作流承载执行。
- [失去实时流式体验] → 首版以可靠完成为优先；页面打开期间可通过轮询或后续增量状态恢复部分实时反馈。
- [会话消息 JSON 字段并发更新冲突] → 当前消息保存在单个 JSON 字段中，后台完成写入前需要重新读取会话并校验当前 turn，避免覆盖用户后续变更。
- [长任务失败不可见] → 失败状态和错误摘要必须持久化，并在聊天页显式展示。
- [多实例重复执行] → 幂等标识和状态更新需要尽量使用条件更新或事务，避免两个请求同时启动同一 turn。
